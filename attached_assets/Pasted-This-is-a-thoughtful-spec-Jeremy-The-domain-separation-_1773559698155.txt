This is a thoughtful spec, Jeremy. The domain separation between Cleo and Miles is clean, and the “agents recommend, user confirms” guardrail is exactly right for a CRM. A few things I’d push back on or flag:
Latency is the real risk here. Every user message hits at minimum three sequential LLM calls: Coach classifies → apprentice(s) execute → Coach synthesizes. That’s 3-4 round trips before the user sees anything meaningful. Even with streaming on the final Coach response, you’re looking at several seconds of dead air while routing and apprentice calls complete behind the scenes. For a mobile chat UI, that’s going to feel sluggish. Worth prototyping the happy path end-to-end before committing to the architecture.
The “truly separate entities” framing is philosophically strong but architecturally expensive. In practice, Coach, Cleo, and Miles are the same model with different system prompts. The separation buys you cleaner prompt boundaries, but you could get 80% of the value with a single call that uses structured tool use — where “route to Cleo’s tools” and “route to Miles’s tools” are just different tool sets the model selects from. That cuts your latency roughly in half and simplifies the orchestration layer significantly. The personality layer can still be applied when formatting the response.
The background insight worker needs a budget ceiling. Running two full LLM scans on a schedule against the whole contact/lead database could get expensive fast, especially as the dataset grows. I’d scope this tightly — define a max record count per scan, a fixed schedule (daily, not hourly), and a cost cap. Start with the simplest heuristic-based checks (overdue follow-ups, leads idle > X days) as plain SQL queries, and only call the LLM for the natural-language framing of those results.
A few smaller things: the spec mentions gpt-5.2 via Replit AI proxy — worth confirming that model string is live and available through that proxy before building against it. The tool definitions for each agent are described conceptually but not as actual function schemas, which is where the real implementation complexity lives. And the conversation context management across three agents needs a clear design — does Cleo see Miles’s prior outputs? Does Coach’s history include the full apprentice exchanges or just summaries?
The core product idea is solid — a CRM that proactively tells you who to call and why, with strategic context layered on top. The question is whether the three-agent ceremony justifies the latency and complexity cost over a single well-prompted agent with good tool routing. I’d suggest building a quick prototype of both approaches against the same test scenarios and comparing response quality and speed before locking in the architecture.​​​​​​​​​​​​​​​​
